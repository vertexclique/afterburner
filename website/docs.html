Afterburner Docs.

A drop-in JavaScript runtime for Rust. Run real Node.js apps with burn, or embed the same engine in your Rust program with one crate. Hard resource limits and fine-grained capability controls are first-class. Every limit, every gate, configurable.

v0.1 · stable MSRV 1.90 2024 edition Apache-2.0

One crate. One binary.
Zero unsafe in your code.

Afterburner ships a burn CLI that runs Node apps unmodified, and an embeddable Rust crate behind a single ergonomic builder. The defaults are safe; the limits are yours to set.

$ cargo add afterburner · Click to copy

#Introduction

Afterburner is a JavaScript runtime built in Rust. The burn binary runs your existing Node apps (burn server.js, burn npm install, burn app.ts), and the embeddable afterburner crate exposes the same engine to your Rust program. Every Node built-in (fs, crypto, http, zlib, child_process, …) is gated by a capability you configure per script.

Memory caps, deterministic fuel metering, an N-worker scheduler, and a fully sandboxed execution mode are the production-grade machinery underneath. The surface you touch is one builder and three methods (register, run, unregister), or just type burn at the shell.

i
Drop-in promise.

Code that runs under node script.js should run unchanged under burn script.js. Where it doesn't, that's a bug. File an issue with a Node.js docs link and a minimal repro.

#Installation

The fastest path: afterburner.sh dispatches by user agent, verifies the SHA-256 sidecar, and drops burn into ~/.local/bin. Pick your shell:

$ Copy

Pin a version with BURN_VERSION=vX.Y.Z · custom install dir with BURN_INSTALL=/usr/local/bin · skip the $PATH update with BURN_INSTALL_NO_PATH=1

Or pull the runtime as a Rust library:

Cargo.toml 
[dependencies]
afterburner = "0.1"

Or build the burn CLI from source:

shell 
$ cargo install afterburner --features bin
$ burn --version
afterburner 0.1.1

The prebuilt binary ships with --features release-cli (every backend, every L3 shadow, TypeScript loader). Nothing external is needed at runtime; SQLite, OpenSSL, libclang are all statically linked in. See Building from source if you want to compile the workspace yourself.

#Your first burn

Five lines. Register a script, hand it some JSON, get JSON back:

rust 
use afterburner::Afterburner;
use serde_json::json;

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let ab = Afterburner::new()?;
    let id = ab.register("module.exports = (d) => d.n + 1")?;
    let out = ab.run(&id, &json!({ "n": 41 }))?;
    assert_eq!(out, json!(42));
    Ok(())
}

Afterburner::new() uses the adaptive mode by default: a fast cold start on the first call, then a fully sandboxed execution path from the second call onward. You don't pick. It picks.

#The Manifold

Every host capability lives behind a Manifold. The library default is Manifold::sealed(), safe to hand untrusted user scripts. Everything else you grant explicitly.

rust 
use afterburner::{Afterburner, Manifold, FsAccess, NetAccess};

let ab = Afterburner::builder()
    .manifold(Manifold {
        fs:  FsAccess::ReadWrite(vec!["/var/data".into()]),
        net: NetAccess::Allow(vec!["api.example.com".into()]),
        crypto: true,
        ..Manifold::sealed()
    })
    .build()?;

!
The CLI defaults differ.

The library default is Manifold::sealed(), which denies everything. The burn CLI defaults to Manifold::open() so Node scripts drop in without flags. Pass --sandbox to flip it back to sealed for untrusted input.

#Fuel, memory & timeouts

Three independent dimensions, all hard caps:

rust 
let ab = Afterburner::builder()
    .fuel(your_op_budget)         // operation count budget
    .memory_bytes(your_memory_cap)  // per-script memory ceiling
    .timeout_ms(your_deadline)      // wall-clock kill switch
    .build()?;
  • Fuel meters JS bytecode operations. A tight loop trips the gauge in deterministic op-time, not wall-clock time, so behavior is reproducible across hosts. Pick whatever bound matches your workload.
  • Memory is enforced by the sandbox allocator. Over-budget allocations return JS-side OutOfMemory, never an OOM kill of the host.
  • Timeout is wall-clock; backstop for blocking work that doesn't burn fuel (e.g. host I/O).
i
Nothing is hardcoded.

The runtime ships sensible defaults, but every limit on this page is a knob you set per build, per call, or per CLI invocation (--fuel, --memory, --timeout, --workers). Pick what fits your workload.

i
Long-running daemons: the pool ceiling.

The WASM-mode pooling allocator pre-reserves a per-instance virtual address range — 1 GiB by default. Daemon-mode workloads (Express, Fastify, etc. that share a single QuickJS Store across many requests) can accumulate state across the whole listener lifetime and eventually touch this ceiling, even though the host has plenty of free RAM. Override with BURN_MAX_LINEAR_MEMORY at startup: BURN_MAX_LINEAR_MEMORY=4G ./burn server.js bumps the per-slot ceiling to 4 GiB (the wasm32 ABI maximum); =128M tightens it. --memory <N> sets a tighter per-call cap below this hard limit and is independent of -A (which only flips capability gates, not resource caps).

i
HTTP servers run across all your CPUs.

When your script binds an HTTP listener (http.createServer().listen(...)), burn serves requests from every CPU the OS gives the process, in parallel. Container CPU limits flow through automatically: docker run --cpus=4 gives you 4-way parallelism, k8s cpu: "4" the same. No knob to set.

This expansion only kicks in when init binds an HTTP listener. Other daemon shapes — timer-only scripts, raw net / tls / dgram servers, scripts that open outbound clients — stay single-shard so init-time side effects don't multiply across workers. BURN_SHARDS=N overrides this for benchmarks or workloads where init is intentionally idempotent.

Per-request state in JavaScript is local to each parallel worker, not global. If a handler increments a counter declared outside the route, you'll get N independent counter streams instead of a single monotonic one — the same model Node's cluster module uses. For state shared across requests, use require('afterburner:state').

Sandboxing is preserved end-to-end: capability gates (--allow-fs / --allow-net / --allow-env / --sandbox) apply identically to every parallel worker. A crash in one request can't take down the others — failed requests return HTTP 500 and the daemon keeps serving. Resource limits stack: BURN_MAX_LINEAR_MEMORY=1G with 16-way parallelism means up to 16 GiB peak memory; size your container accordingly.

#Adaptive execution (Flying Start)

The default mode boots fast, then settles into the safer execution path without you noticing:

Cold start
Sub-millisecond ignition path optimised for first-call latency.
Warm path
Fully sandboxed execution with bytecode cached across calls.
Hot path
Pooled instances, fuel-precharged, ~µs steady-state per call.

If you want to pin a single mode, the builder takes .mode(Mode::Fast) for the cold-start path or .mode(Mode::Sandboxed) for the locked-down path. Mode::Adaptive is the default.

#Multi-threaded scheduling

One flag turns a single-threaded runtime into an N-worker scheduler. Bounded per-worker queues plus a global injector; token-bucket admission keeps the system responsive under load; steal-when-idle keeps cores busy without thrashing locality. The width is yours to choose.

rust 
let ab = Afterburner::builder()
    .threaded(your_worker_count)    // any N >= 1
    .route_by(RouteBy::HashKey)    // pin sessions to a worker
    .build()?;

for row in rows {
    ab.submit(&id, &row, row.session_key())?;
}
ab.drain().await?;            // graceful shutdown

#Cross-invocation state

The custom afterburner:state module lets scripts share data across calls. The default InMemoryStateStore uses a lock-free HopscotchMap with AtomicI64 counters. Embedders can swap their own (Redis, SQLite, anything) by implementing the StateStore trait.

rust · host side 
use afterburner::{Afterburner, InMemoryStateStore};

let store = InMemoryStateStore::shared();
let ab = Afterburner::builder()
    .state_store(store.clone())
    .build()?;
javascript · script side 
const state = require('afterburner:state');

state.setJSON('lastSeen', Date.now());
const hits = state.increment('hits');          // atomic i64
const seen = state.get('lastSeen');
const keys = state.listKeys('session:');          // prefix scan

module.exports = () => ({ hits, seen });

#Embedder extension points

Three pluggable seams on the Rust side, each with a sensible default. Swap your own when the default is too tight.

BurnCacheBackend

Compiled scripts are content-addressed by SHA-256 of the source. Implement BurnCacheBackend to share that cache across processes or hosts (Redis, S3, SQLite, NATS). The default InProcessCacheBackend is a lock-free map per Afterburner instance.

rust 
use afterburner::{Afterburner, BurnCacheBackend, Result};
use std::sync::Arc;

struct RedisBackend { client: redis::Client }
impl BurnCacheBackend for RedisBackend {
    fn fetch(&self, hash: &[u8; 32]) -> Result<Option<String>> { /* GET */ }
    fn publish(&self, hash: &[u8; 32], source: &str) -> Result<()> { /* SETEX */ }
}

let ab = Afterburner::builder()
    .cache_backend(Arc::new(RedisBackend { client }))
    .build()?;

See examples/cache-backend-sqlite for a working SQLite-WAL coordinator that two Afterburner instances share scripts through.

HostContext

Custom callbacks scripts can reach via require('afterburner:host'). Useful for streaming UDFs (emit_row), structured logging (log), and reading host-resolved configuration (get_env) without granting raw process.env access.

rust 
use afterburner::{Afterburner, HostContext, LogLevel};
use serde_json::Value;
use std::sync::Arc;

struct AppHost { config: AppConfig, sink: Sender<Value> }
impl HostContext for AppHost {
    fn log(&self, level: LogLevel, msg: &str) {
        tracing::info!(?level, msg);
    }
    fn get_env(&self, key: &str) -> Option<String> {
        self.config.resolve(key)
    }
    fn emit_row(&self, row: Value) {
        let _ = self.sink.try_send(row);
    }
}

let ab = Afterburner::builder()
    .host_context(Arc::new(AppHost::new(cfg, tx)))
    .build()?;

FlowEngine (multi-module bundles)

For pipelines that span multiple ES modules. Enable the flow feature, then call .flow() on the builder and register_bundle(entry, modules) to compile a graph of files at once.

rust 
let ab = Afterburner::builder().flow().build()?;
let id = ab.register_bundle("./pipeline.mjs", &[
    ("./normalize.mjs".into(), normalize_src),
    ("./enrich.mjs".into(),    enrich_src),
])?;
let out = ab.run(&id, &trigger)?;
i
StateStore is the fourth seam.

Covered in Cross-invocation state above. The default InMemoryStateStore uses a lock-free HopscotchMap with AtomicI64 counters; swap your own to back require('afterburner:state') with Redis, SQLite, or anything else.

#The burn CLI

A Deno-style command-line runtime. CLI default is Manifold::open() so existing Node scripts drop in without flags. Pass --sandbox to flip back to deny-by-default and grant with --allow-*.

shell 
$ burn ./script.js                          # run a file
$ burn -e 'module.exports = () => 42'     # eval inline
$ echo '{"n":21}' | burn thrust transform.js  # UDF mode (stdin JSON in, stdout JSON out)
$ burn bench perf.js --iters 10000 --workers 8
$ burn repl                                   # interactive
$ burn check app.ts                           # compile + parse, exit 0/1

# Pass-through to the Node ecosystem. Nested node calls route home.
$ burn npm install express
$ burn pnpm run dev
$ burn npx tsc --init
$ burn yarn build
$ burn bun run start

# Capability grants
$ burn --sandbox \
       --allow-fs=./data,/tmp/jobs \
       --allow-net=api.example.com,*.trusted.io \
       --allow-env=NODE_ENV,API_KEY \
       script.js
$ burn -A runall.js                            # grant everything

#Sandbox security omissions

The runtime intentionally refuses the following APIs because they would let user JS escape the capability sandbox. Each refusal throws a typed, Node-shaped error so existing code fails fast rather than mis-running.

APIRefusal modeWhy
process.chdir(dir)throws Error('process.chdir is not supported')Working-directory mutation breaks the per-call Manifold fs allow-list — paths resolved relative to cwd would migrate outside the granted roots.
process.setuid / setgid / seteuid / setegid / setgroupsthrows {code: 'EPERM'}Identity drop/elevation bypasses the host's effective UID. The runtime cannot grant the kernel something it doesn't have itself, and a successful syscall here would lie to the caller.
new Worker(code, { eval: true })throws Error("eval: true not supported")Compiling an arbitrary string into a fresh JS realm sidesteps the per-script transpile + fs allow-list checks. new Worker(path) (real file on disk) remains the supported shape and gives the same parallelism.
process.binding(name)throws Error("process.binding is not supported")Node's private internals API. Userland code that depends on it is non-portable; legitimate use cases all have a node:<module> equivalent.
Atomics.wait (sync) on non-shared buffersthrows TypeError("requires a SharedArrayBuffer-backed view")Blocking on a non-shared buffer is a guaranteed deadlock in the single-threaded JS realm. The shared-buffer path is real (futex-backed cross-process) and goes through the host instead of trapping the realm.
Debugger.setBreakpointByUrl when BURN_DEBUGGER_INSTRUMENT is unsetregisters the breakpoint but never firesAlways-on source-level instrumentation adds one fn-call per statement. Gating it behind an env var keeps the production hot path zero-overhead; the inspector listener still accepts the command so external tools don't see a hard "not implemented".
setTimeout(fn, delay>0) outside daemon modethrows Error("not supported in this sandbox")One-shot script mode (burn script.js with no listeners) has no event loop to drive timers. setTimeout(fn, 0) still fires via the microtask queue; non-zero needs daemon mode (which the CLI enters automatically once a listener or worker is created).
require('./x.ts') without the ts cargo featurethrows Error("Loading 'x.ts' requires the `ts` cargo feature")TypeScript strip-transpile pulls in oxc (~3 MiB of compile cost). The feature is opt-in so embedders that only run plain JS don't pay it. Build with --features ts to enable.
node:quic / http3 without the http3 cargo featurethrows on QuicEndpoint constructionQUIC pulls in quinn + h3 (~3 MiB). Build with --features http3.
Native .node addons (require('./binding.node'))throws at require-timeNative addons bypass the WASM sandbox entirely — they're arbitrary code from disk linked into the host process. For the popular ones (bcrypt, argon2, jsonwebtoken, sqlite3, sharp) burn ships pure-Rust shadows under the same npm-facing API.
✓
Custom modules

require('afterburner:state') for a cross-invocation key/value store with atomic increment_i64. require('afterburner:host') for the HostContext bridge (log, emit_row, read_column, get_env). Both implicitly available when the embedder installs the corresponding trait.

#Environment variables

VariableDefaultPurpose
AFTERBURNER_LOGwarnLevel: off · error · warn · info · debug · trace
AFTERBURNER_LOG_FORMATtextReporter format: text (stderr) or json (stdout NDJSON)
AFTERBURNER_REBUILD_PLENUMunsetSet to 1 to regenerate generated/plenum_bundle.js.
BURN_MAX_LINEAR_MEMORY1G (1 GiB)Per-instance WASM linear-memory ceiling enforced by Wasmtime's pooling allocator. Accepts plain bytes or a K/M/G suffix (e.g. 4G = 4 GiB, 128M = 128 MiB). Hard-clamped to 4 GiB (the wasm32 ABI limit). Bump it for long-running daemon-mode workloads (Express, Fastify, etc.) that accumulate per-request state across many calls; lower it when running many concurrent instances on small hosts. --memory <N> on the CLI sets a tighter per-call cap below this hard limit and is independent of -A.

BURN_INSTALL_NO_PATHunsetSet to 1 to skip the bootstrap installer's automatic $PATH update (it normally appends an export PATH=... line to your shell rc — ~/.bashrc, ~/.zshrc, fish config, or ~/.profile — or sets the User Path environment variable on Windows). Use this if you manage $PATH yourself via dotfiles, chezmoi, or a NixOS / nix-home / Guix configuration.
BURN_QUIETunsetSet to 1 to suppress the first-run capability banner and other non-essential stderr notices. Same as --quiet / -q.
BURN_SHARDSauto (matches available CPUs)Number of parallel workers the daemon uses to serve HTTP. Defaults to the number of CPUs your container or host gives the process, which is usually what you want — set this only for testing (BURN_SHARDS=1 reverts to single-worker behavior), benchmarking different parallelism levels without changing the container, or pinning a specific count on a CI runner. Accepted range 1–128; values outside that range or non-numeric input fall back to auto with a warning. Setting it higher than the number of available CPUs is allowed but discouraged — workers compete for cores they don't have, which slows you down rather than speeds you up.
BURN_SHIM_DEPTH0 (incremented per shim hop, capped at 8)Pass-through recursion guard. When burn is invoked on a non-burn binary (burn node script.js, burn npm test, etc.) it forwards to the underlying tool and increments this counter — hitting 8 aborts with a typed error instead of fork-bombing. Operators rarely set this manually; it exists so the shim layer can refuse cycles introduced by a misconfigured PATH or a wrapper script that re-invokes burn.
BURN_WORKER_DEPTH0 (incremented per spawned worker, capped at 8)worker_threads recursion guard (mirrors BURN_SHIM_DEPTH for the worker model). Each spawned new Worker(...) child inherits the parent's depth + 1; hitting the ceiling refuses spawn with E_DEPTH instead of letting a runaway script exhaust system resources. Together with WorkerConfig::max_concurrent (default 32) and the 16 MiB per-frame cap, this bounds the worst case a hostile or buggy script can do.

#Building from source

The pre-built burn binary needs nothing external at runtime. To compile the workspace yourself you need a recent Rust toolchain (2024 edition, MSRV 1.90+) and the system C toolchain. That's it. cargo build is reproducible and offline; nothing is downloaded mid-build beyond crate dependencies.

shell 
$ git clone https://github.com/vertexclique/afterburner
$ cd afterburner
$ cargo build --release --features bin
$ ./target/release/burn --version
i
Reproducible by design.

All execution-engine artifacts are committed in-tree and pulled via include_bytes!. cargo build is fully offline once dependencies are resolved. No extra toolchains on every host.
