Afterburner Docs.

A drop-in JavaScript runtime for Rust. Run real Node.js apps with burn, or embed the same engine in your Rust program with one crate. Hard resource limits and fine-grained capability controls are first-class. Every limit, every gate, configurable.

v0.1 · stable MSRV 1.90 2024 edition Apache-2.0

One crate. One binary.
Zero unsafe in your code.

Afterburner ships a burn CLI that runs Node apps unmodified, and an embeddable Rust crate behind a single ergonomic builder. The defaults are safe; the limits are yours to set.

$ cargo add afterburner · Click to copy

#Introduction

Afterburner is a JavaScript runtime built in Rust. The burn binary runs your existing Node apps (burn server.js, burn npm install, burn app.ts), and the embeddable afterburner crate exposes the same engine to your Rust program. Every Node built-in (fs, crypto, http, zlib, child_process, …) is gated by a capability you configure per script.

Memory caps, deterministic fuel metering, an N-worker scheduler, and a fully sandboxed execution mode are the production-grade machinery underneath. The surface you touch is one builder and three methods (register, run, unregister), or just type burn at the shell.

i
Drop-in promise.

Code that runs under node script.js should run unchanged under burn script.js. Where it doesn't, that's a bug. File an issue with a Node.js docs link and a minimal repro.

#Installation

Add the facade crate to your Cargo.toml:

Cargo.toml 
[dependencies]
afterburner = "0.1"

Or install the burn CLI:

shell 
$ cargo install afterburner --features bin
$ burn --version
afterburner 0.1.0

Nothing external is needed at runtime; the runtime is fully self-contained in the binary. See Building from source if you want to compile the workspace yourself.

#Your first burn

Five lines. Register a script, hand it some JSON, get JSON back:

rust 
use afterburner::Afterburner;
use serde_json::json;

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let ab = Afterburner::new()?;
    let id = ab.register("module.exports = (d) => d.n + 1")?;
    let out = ab.run(&id, &json!({ "n": 41 }))?;
    assert_eq!(out, json!(42));
    Ok(())
}

Afterburner::new() uses the adaptive mode by default: a fast cold start on the first call, then a fully sandboxed execution path from the second call onward. You don't pick. It picks.

#The Manifold

Every host capability lives behind a Manifold. The library default is Manifold::sealed(), safe to hand untrusted user scripts. Everything else you grant explicitly.

rust 
use afterburner::{Afterburner, Manifold, FsAccess, NetAccess};

let ab = Afterburner::builder()
    .manifold(Manifold {
        fs:  FsAccess::ReadWrite(vec!["/var/data".into()]),
        net: NetAccess::Allow(vec!["api.example.com".into()]),
        crypto: true,
        ..Manifold::sealed()
    })
    .build()?;

!
The CLI defaults differ.

The library default is Manifold::sealed(), which denies everything. The burn CLI defaults to Manifold::open() so Node scripts drop in without flags. Pass --sandbox to flip it back to sealed for untrusted input.

#Fuel, memory & timeouts

Three independent dimensions, all hard caps:

rust 
let ab = Afterburner::builder()
    .fuel(your_op_budget)         // operation count budget
    .memory_bytes(your_memory_cap)  // per-script memory ceiling
    .timeout_ms(your_deadline)      // wall-clock kill switch
    .build()?;
  • Fuel meters JS bytecode operations. A tight loop trips the gauge in deterministic op-time, not wall-clock time, so behavior is reproducible across hosts. Pick whatever bound matches your workload.
  • Memory is enforced by the sandbox allocator. Over-budget allocations return JS-side OutOfMemory, never an OOM kill of the host.
  • Timeout is wall-clock; backstop for blocking work that doesn't burn fuel (e.g. host I/O).
i
Nothing is hardcoded.

The runtime ships sensible defaults, but every limit on this page is a knob you set per build, per call, or per CLI invocation (--fuel, --memory, --timeout, --workers). Pick what fits your workload.

#Adaptive execution (Flying Start)

The default mode boots fast, then settles into the safer execution path without you noticing:

Cold start
Sub-millisecond ignition path optimised for first-call latency.
Warm path
Fully sandboxed execution with bytecode cached across calls.
Hot path
Pooled instances, fuel-precharged, ~µs steady-state per call.

If you want to pin a single mode, the builder takes .mode(Mode::Fast) for the cold-start path or .mode(Mode::Sandboxed) for the locked-down path. Mode::Adaptive is the default.

#Multi-threaded scheduling

One flag turns a single-threaded runtime into an N-worker scheduler. Bounded per-worker queues plus a global injector; token-bucket admission keeps the system responsive under load; steal-when-idle keeps cores busy without thrashing locality. The width is yours to choose.

rust 
let ab = Afterburner::builder()
    .threaded(your_worker_count)    // any N >= 1
    .route_by(RouteBy::HashKey)    // pin sessions to a worker
    .build()?;

for row in rows {
    ab.submit(&id, &row, row.session_key())?;
}
ab.drain().await?;            // graceful shutdown

#Cross-invocation state

The custom afterburner:state module lets scripts share data across calls. The default InMemoryStateStore ships with the workspace; embedders can plug in their own (Redis, SQLite, …) by implementing the StateStore trait.

rust · host side 
use afterburner_core::InMemoryStateStore;
use afterburner_wasi::{WasmCombustor, WasmConfig};

let store = InMemoryStateStore::shared();
let combustor = WasmCombustor::new(WasmConfig {
    state_store: Some(store.clone()),
})?;
javascript · script side 
const state = require('afterburner:state');

state.setJSON('lastSeen', Date.now());
const n = state.increment('hits');

module.exports = () => ({ hits: n });

#The burn CLI

A Deno-style command-line runtime. Deny by default, grant with --allow-* flags.

shell 
$ burn ./script.js                          # run a file
$ burn -e 'module.exports = () => 42'     # eval inline
$ echo '{"n":21}' | burn thrust transform.js  # UDF mode
$ burn bench perf.js --iters 10000 --workers 8
$ burn repl                                   # interactive

# Deno-style capability grants
$ burn --allow-net=api.example.com,*.trusted.io script.js
$ burn --allow-fs=/tmp,/var/data etl.js
$ burn --allow-env=HOME,PATH launcher.js
$ burn -A runall.js                            # grant everything

Flag reference

FlagEffect
-e <src>Evaluate inline source instead of loading a file.
thrust <file>UDF mode: stdin → JSON → script → JSON → stdout, line-delimited.
bench <file>Throughput smoke test with iteration + worker counts.
replInteractive evaluator with history and tab-complete.
--allow-net=<hosts>Comma list, wildcards allowed (*.trusted.io).
--allow-fs=<roots>Comma list of read-write roots. --allow-fs-read for read-only.
--allow-env=<keys>Comma list of environment variable keys.
--sandboxFlip CLI default from open() to sealed().
-AGrant everything. Useful for trusted scripts; never for untrusted input.

#Building from source

The pre-built burn binary needs nothing external at runtime. To compile the workspace yourself you need a recent Rust toolchain (2024 edition, MSRV 1.90+) and the system C toolchain. That's it. cargo build is reproducible and offline; nothing is downloaded mid-build beyond crate dependencies.

shell 
$ git clone https://github.com/vertexclique/afterburner
$ cd afterburner
$ cargo build --release --features bin
$ ./target/release/burn --version
i
Reproducible by design.

All execution-engine artifacts are committed in-tree and pulled via include_bytes!. cargo build is fully offline once dependencies are resolved. No extra toolchains on every host.
